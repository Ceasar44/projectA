# 后端队列、异步任务与缓存说明

说明：

- 本文基于当前后端代码扫描整理，重点查看消息队列、异步任务、缓存、事件总线、后台任务、outbox 等实现。
- 当前项目没有接入 Celery、RQ、RabbitMQ、Kafka、Redis Queue 等独立消息队列服务。
- `REDIS_URL` 存在于配置中，但当前后端没有发现 Redis 客户端或实际 Redis 缓存/队列调用。

## 一、已实际使用的机制

### 1. 进程内实时事件总线

位置：

- `backend/app/infrastructure/realtime/__init__.py`
- `backend/app/api/v1/realtime.py`
- `backend/app/api/v1/conversations.py`
- `backend/app/api/v1/chat.py`

用途：

- 用于 SSE 实时推送会话消息、会话更新、输入中状态、通知等事件。
- `RealtimeEventBus` 维护一个进程内订阅者表：
  - `_subscribers: dict[str, set[EventCallback]]`
  - key 是频道名，例如 `global`、`conversation:{conversation_id}`、`user:{user_id}`。
  - value 是回调函数集合。

数据结构：

- `EventType`：事件类型枚举。
- `EventPayload`：事件载荷，包含 `type`、`data`、`timestamp`、`conversation_id`。
- `RealtimeEventBus._subscribers`：内存订阅表。

典型流程：

1. 前端调用 `GET /api/realtime` 建立 SSE 连接。
2. 接口内部创建当前连接自己的 `asyncio.Queue[EventPayload]`。
3. 通过 `event_bus.subscribe(channel, on_event)` 注册回调。
4. 业务代码调用 `event_bus.emit_new_message`、`emit_conversation_update` 等方法发布事件。
5. 回调将事件放入该 SSE 连接的 `asyncio.Queue`。
6. SSE 生成器从队列取事件并推送给前端。

重要特性：

- 这是进程内事件总线，不是跨进程/跨机器消息队列。
- 服务重启后订阅关系丢失。
- 多 worker 部署时，不同 worker 之间不会自动同步事件。
- 没有持久化，也没有消费确认或重放机制。

### 2. SSE 连接内的 `asyncio.Queue`

位置：

- `backend/app/api/v1/realtime.py`

用途：

- 每个 SSE 连接创建一个 `asyncio.Queue`，作为该连接的事件缓冲。

相关代码行为：

- `queue: asyncio.Queue[EventPayload] = asyncio.Queue()`
- 事件回调中使用 `queue.put_nowait(event)`。
- SSE 循环中使用 `asyncio.wait_for(queue.get(), timeout=30.0)`。
- 超时后发送 heartbeat。

说明：

- 这个队列只存在于单个 HTTP/SSE 连接生命周期内。
- 它不是全局任务队列，也不做持久化。

### 3. FastAPI `BackgroundTasks`

位置：

- `backend/app/rag/api/v1/documents.py`
- `backend/app/rag/services/document_service.py`
- `backend/app/rag/services/job_service.py`

用途：

- RAG 文档上传或从分类启动切片后，使用 FastAPI `BackgroundTasks` 在响应返回后继续执行文档处理流水线。

典型入口：

- `POST /api/rag/v1/documents/upload`
- `POST /api/rag/v1/documents/start-chunking/{category_id}`
- `POST /api/rag/v1/documents/start-chunking-excel/{category_id}`

典型流程：

1. 上传文件或选择分类文件。
2. 后端校验文件、上传 OSS、创建 `knowledge_file`。
3. 创建 `knowledge_job`，初始状态通常是 `pending`。
4. 调用 `background_tasks.add_task(run_pipeline, ...)`。
5. 接口先返回 `job_id`、`file_id` 等信息。
6. 后台任务调用 `run_job_pipeline`。
7. 任务下载文件、解析文本/图片、生成切片、写入 `knowledge_chunk`，并更新 `knowledge_job` 和 `knowledge_file` 状态。

说明：

- 这是 FastAPI/Starlette 的请求后后台任务机制，不是独立 worker。
- 任务随当前 Web 进程执行。
- 进程崩溃时任务不会自动恢复。
- 当前没有看到基于 Redis/Celery 的分布式任务队列。

### 4. `asyncio.to_thread` 线程卸载

位置：

- `backend/app/rag/services/job_service.py`

用途：

- 在 RAG 文档处理和向量写入中，把阻塞型操作放到线程中执行，避免长时间阻塞事件循环。

使用场景：

- 下载 OSS 文件：`await asyncio.to_thread(_download_file, oss_key)`
- 解析 PDF/Word/Excel/文本：`await asyncio.to_thread(_parse_text_mode, ...)`
- 图片模式解析：`await asyncio.to_thread(_parse_image_mode, ...)`
- 向 Milvus 分批 upsert：`await asyncio.to_thread(_upsert_chunks_in_batches, ...)`

说明：

- 这属于异步服务中的阻塞任务隔离，不是消息队列。
- 执行仍在当前后端进程内完成。

### 5. 配置缓存 `functools.lru_cache`

位置：

- `backend/app/core/config.py`

用途：

- `get_settings()` 使用 `@lru_cache` 缓存 Pydantic Settings 实例，避免每次读取配置都重新解析环境变量和 `.env`。

相关代码：

```python
@lru_cache
def get_settings() -> Settings:
    return Settings()
```

说明：

- 这是进程内函数结果缓存。
- 缓存只对当前 Python 进程有效。
- `.env` 修改后，运行中的进程不会自动刷新，需要重启进程或显式清理缓存。

## 二、已有抽象或预留，但当前未接入主业务链路的机制

### 1. 内存任务队列接口

位置：

- `backend/app/infrastructure/tasks/ports.py`
- `backend/app/infrastructure/tasks/workflows.py`

对象：

- `TaskQueue`：任务队列协议，定义 `enqueue(task_name, payload)`。
- `InMemoryTaskQueue`：内存实现，内部使用 `self.jobs: list[dict]` 保存任务。
- `WorkflowDispatcher`：把领域事件映射成任务名，再调用队列入队。

当前映射：

| 事件名 | 任务名 |
| --- | --- |
| `conversation.created` | `sync_conversation_metrics` |
| `message.received` | `evaluate_automation_rules` |
| `webhook.delivery.failed` | `retry_webhook_delivery` |
| 其他 | `noop` |

说明：

- 这是任务队列抽象和内存样例实现。
- 当前代码扫描中没有发现它被主路由、服务层或启动流程实际实例化使用。
- `InMemoryTaskQueue` 只追加到内存 list，没有 worker、重试、持久化、并发控制。

### 2. Domain Event Outbox 抽象

位置：

- `backend/app/infrastructure/events/types.py`
- `backend/app/infrastructure/events/outbox.py`

对象：

- `DomainEvent`：领域事件对象，包含 `name`、`payload`、`occurred_at`、`event_id`。
- `OutboxMessage`：outbox 消息对象，包含 `topic`、`payload`、`event_id`。
- `OutboxRepository`：发布协议。
- `InMemoryOutboxRepository`：内存 outbox 实现。

说明：

- 这是 outbox 模式的预留抽象。
- 当前没有数据库 outbox 表。
- 当前代码扫描中没有发现主业务服务实际调用 `InMemoryOutboxRepository.publish`。
- 因此目前不能视为已接入的可靠消息 outbox，只能视为架构预留。

### 3. Webhook 待重试记录

位置：

- `backend/app/infrastructure/webhooks/delivery.py`
- `backend/app/domain/webhooks/service.py`
- `backend/app/infrastructure/db/models/operations.py`

相关表：

- `webhook_delivery`

用途：

- Webhook 投递失败时，记录 `status`、`attempts`、`last_error`、`next_retry_at`。
- `WebhookDeliveryService.process_pending_retries()` 可以扫描到期的 pending 记录并重试。

说明：

- 当前确实有“待重试记录”和“处理待重试记录”的函数。
- 但代码扫描中没有发现定时调度器或后台 worker 自动调用 `process_pending_retries()`。
- 因此当前更像是数据库状态记录和手动/预留重试机制，不是完整异步队列。

## 四、结论

当前后端使用过以下数据结构或机制：

| 类型 | 是否使用 | 说明 |
| --- | --- | --- |
| 进程内事件总线 | 是 | `RealtimeEventBus` 用于 SSE 实时推送。 |
| `asyncio.Queue` | 是 | 每个 SSE 连接内部使用一个队列缓冲事件。 |
| FastAPI `BackgroundTasks` | 是 | RAG 文档上传/切片任务在响应后继续处理。 |
| `asyncio.to_thread` | 是 | RAG 文档解析、OSS 下载、向量写入等阻塞操作卸载到线程。 |
| `lru_cache` | 是 | 缓存 `get_settings()` 配置对象。 |
| 内存任务队列 | 预留 | 有 `TaskQueue` 和 `InMemoryTaskQueue`，但未接入主业务链路。 |
| Outbox | 预留 | 有内存 outbox 抽象，但未持久化、未接入主业务链路。 |
| Webhook 重试队列 | 部分预留 | 有待重试状态和处理函数，但未发现自动调度器。 |
| Redis | 否 | 只有 `REDIS_URL` 配置，未发现实际使用。 |
| Celery/RQ/RabbitMQ/Kafka | 否 | 未发现相关依赖或调用。 |

总体判断：

- 当前项目主要依靠“进程内机制”完成实时推送和后台处理。
- 还没有完整的分布式消息队列、持久化任务队列或 Redis 缓存体系。
- 如果未来需要多 worker 部署、任务可靠重试、跨进程实时广播或可恢复后台任务，应考虑把实时事件、RAG 后台任务和 Webhook 重试迁移到 Redis/Celery/RQ/Arq/Kafka 等外部基础设施。
