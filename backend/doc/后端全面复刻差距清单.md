# Python FastAPI Backend 与原 TS Backend 全面对比与剩余工作清单

更新时间: 2026-04-15

## 1. 本次检查范围

本次对比基于以下代码进行人工核对:

- TS API 路由: `src/app/api/**/route.ts`
- TS 核心工作流: `src/lib/ai/engine.ts`, `src/lib/ai/tools.ts`, `src/lib/automation.ts`, `src/lib/customer-resolver.ts`, `src/lib/campaigns.ts`, `src/lib/flow-builder.ts`
- TS 数据模型: `prisma/schema.prisma`
- Python API 路由: `backend/app/api/v1/*.py`
- Python 关键实现: `backend/app/domain/**`, `backend/app/infrastructure/db/**`, `backend/app/infrastructure/integrations/**`

结论先说:

- `backend` 现在已经覆盖了绝大多数 TS API 路径，且大部分高频 CRUD 契约已经被收敛到接近 1:1。
- 但它还没有“完全复刻”原 TS 后端逻辑。
- 当前最大的差距已经不再是“有没有接口”，而是“接口背后的工作流、副作用、外部集成和数据库语义是否完全等价”。

- `customers`
- `tickets`
- `knowledge`
- `team`

- `activity`

这些模块当前已经具备:

## 3. 数据库层仍需补完的内容

### 3.1 默认数据库仍不是 PostgreSQL

原 TS 侧 Prisma schema 明确以 PostgreSQL 为目标。

Python backend 虽然支持 PostgreSQL 连接，但当前默认配置仍然不是 PostgreSQL 生产形态。

要完全复刻，仍需:

- 将运行默认环境切到 PostgreSQL
- 补齐并验证 Alembic 初始迁移
- 用 PostgreSQL 实际跑通全部模型与索引
- 确认 JSON、时间戳、唯一约束、级联删除行为与 Prisma 一致

### 3.2 表覆盖还不是 100%

Prisma schema 中的这些表，Python 侧仍存在未完全等价的情况:

- `Schedule`
  - TS schema 中存在
  - Python 当前未见完整模型/API/业务接入
- `Tag` / `ConversationTag`
  - Python 已有模型
  - 但业务流程接入仍不完整，尤其是会话更新、宏执行、详情与列表的一致性仍需全面核对

### 3.3 索引、约束、关系语义还需要一次数据库级校验

尤其是以下点:

- `ConversationTag @@unique([conversationId, tagId])`
- `WebhookDelivery` 的重试相关查询索引
- `Conversation.createdAt/updatedAt` 与导出查询的时间过滤
- `CallLog.callSid` 唯一性
- `Schedule` 缺失导致的团队排班能力缺口

## 4. API 层仍需补完的内容

### 4.1 路径覆盖虽高，但仍有“存在接口，不代表完全等价”

主要问题不再是缺路由，而是:

- 某些路由只是“可联调实现”
- 某些路由的执行副作用弱于 TS
- 某些路由的内部错误处理和边界行为仍未逐条对齐

### 4.2 需要重点复核的 API 组

#### `chat`

这是当前最关键的未完全复刻模块。

虽然 Python 版现在已经具备:

- 新建会话
- 落消息
- 基础 knowledge 注入
- 工具调用循环
- 情绪/意图/升级基础逻辑

但与 TS 相比仍需进一步补齐:

- 完整复刻 `src/lib/ai/engine.ts` 的 prompt 构造
- 完整复刻 `guardrails` 规则与升级条件
- 完整复刻工具调用后的递归上下文组织
- 完整复刻置信度计算逻辑
- 完整复刻工具调用失败时的降级文案
- 完整复刻消息写入、会话 metadata 更新与状态迁移顺序

#### `knowledge/test`

当前 Python 已补错误分支和返回结构，但仍不是 TS 的真实 AI 调用等价实现。

要完全复刻，仍需:

- 真正调用配置中的 AI provider
- 真实构造 knowledge context
- 从模型响应中解析 `---SOURCES---`
- 将 source index 映射为知识条目

#### `realtime`

Python 版已支持 SSE，但仍比 TS 多出额外能力和事件包装字段。

要完全复刻，仍需:

- 只保留 TS 当前使用的订阅语义
- 统一事件数据包格式
- 对齐 `event_bus` 事件对象到 TS `subscribe(channel, event)` 的实际结构

#### `webhooks`

Python 版现在已经支持:

- CRUD
- delivery 列表
- retry
- test

但仍需:

- 对齐 TS 侧状态命名
- 明确 `success/failed/pending` 与 TS 的 delivery 状态语义
- 验证 retry 的请求体、错误码、幂等行为
- 验证 test 与 real trigger 的行为是否完全一致

#### `channels`

Python 版当前主要是“配置 + 状态 + 联调桩 + 电话链路”。

要完全复刻，仍需:

- SMS
- Telegram
- Email
- WhatsApp
- Phone

这些渠道各自的真实收发链路，而不只是配置状态切换。

## 5. 业务工作流层的核心差距

这是离“完全复刻”最近但也最关键的部分。

### 5.1 AI 工具链还不是完全等价

原 TS `src/lib/ai/tools.ts` 的这些动作都带业务副作用:

- `create_ticket`
- `assign_to_person`
- `send_internal_email`
- `get_customer_history`
- `schedule_followup`
- `trigger_webhook`

Python 版虽然已具备同名工具，但还不是全部达到 TS 等价:

- `create_ticket`
  - 基本有
  - 但部门匹配、响应文案、落库细节还需逐项核对
- `assign_to_person`
  - 基本有
  - 但匹配策略、状态变更、返回内容需与 TS 再逐项比对
- `send_internal_email`
  - Python 已做 SMTP 发送
  - 仍需验证超时、TLS、认证失败、错误信息是否与 TS 一致
- `get_customer_history`
  - 已有基础实现
  - 但跨渠道历史查找仍受客户归并能力影响
- `schedule_followup`
  - 当前仍是占位式成功返回
  - TS 也只是轻实现，但 Python 仍未形成真正可执行 follow-up 机制
- `trigger_webhook`
  - 当前更接近“排队/成功返回”
  - 还未完整复刻 TS 的真实外发行为与错误处理

### 5.2 自动化规则与消息链路仍需彻底复核

TS `evaluateRules` 的核心语义是:

- 新消息到达时触发
- 命中规则后返回动作列表
- 异步递增 `triggerCount`

Python 当前已经接上消息链路，但还需要确认:

- 所有进入点都触发自动化，而不仅是部分入口
- `triggerCount` 行为与 TS 一致
- 宏执行、副作用和实时更新顺序一致
- 各类规则条件字段是否与前端配置保持一致

### 5.3 客户归并逻辑仍需最终确认

Python `resolve_customer` 已经比早期版本更接近 TS，但要完全复刻，还需要确认:

- email/phone/whatsapp 的匹配优先级与 TS 完全一致
- 归一化手机号规则与 TS 完全一致
- 跨字段 fallback 行为完全一致
- 回填空字段行为完全一致
- “Unknown” 名称被更好名称替换的条件完全一致

这块非常重要，因为它直接影响:

- 客户历史查询
- 会话归属
- AI 上下文
- CRM 列表准确性

### 5.4 会话标签工作流还未完全闭环

虽然 Python 已引入 `Tag` / `ConversationTag` 模型，但要真正与 TS 完全一致，还要确认:

- `PUT /api/conversations/{id}` 中 `tagIds` 更新逻辑
- 会话列表的 `tags`
- 会话详情的 `tags`
- 宏执行中的 tag action
- 导出中的 tag 结构
- 客户详情中会话列表是否带统一 tag 结构

如果这部分不再统一校验，后面很容易出现:

- 页面 A 能写 tag
- 页面 B 读不到 tag
- 导出/报表结构不一致

### 5.5 Campaign / Flow 目前仍偏“配置和校验”，不是完整运行态

当前 Python 已经补了:

- Campaign segment 匹配
- Flow validate

但要完全复刻，还需要:

- Campaign 真正执行发送动作
- Flow 运行时执行引擎
- Flow 节点 action 的落地执行
- Flow 与 chat/conversation 的集成
- Campaign 的计划任务、状态迁移、发送统计

TS 当前本身也不是完整的营销发送平台，但 Python 若要称为“完全复刻”，至少要做到与 TS 同等的运行态边界。

## 6. 外部集成层仍需补完的内容

### 6.1 OpenAI / AI Provider

要完全复刻，仍需:

- 按 TS 配置字段完整接通 provider
- 对齐 model/maxTokens/temperature 逻辑
- 对齐工具调用与 completion 调用方式
- 对齐 provider 不可用时的返回文案

### 6.2 SMTP / IMAP

当前 SMTP 基本有，但还需:

- IMAP 接收链路
- Email listener 生命周期
- 与 channel status 的真实联动
- 收件转 conversation 的完整链路

### 6.3 WhatsApp / Telegram / SMS / Phone

当前状态:

- Phone: 已有基础 webhook/TwiML 流程
- WhatsApp: 主要还是 QR/status 联调能力
- Email: 主要还是连接状态能力
- SMS/Telegram: 主要还是占位兼容

要完全复刻，仍需:

- 真正的收消息入口
- 真正的发消息执行
- 渠道状态同步
- 错误处理和重试
- 与 conversation/customer 的自动关联

### 6.4 Webhook delivery worker

当前已有管理和测试能力，但要完全复刻，还需要:

- 真正的异步 delivery worker
- 失败重试调度
- `nextRetryAt` 的执行器
- 可恢复的投递状态机

## 7. 文档、迁移和运维层仍需补完的内容

### 7.1 Alembic 迁移需要正式化

当前还不能把 Python backend 视为“真正可替代 TS 数据层”的一个重要原因，是迁移体系还没彻底收口。

要完全复刻，建议补齐:

- 初始 schema migration
- 后续增量 migration 策略
- PostgreSQL 本地/测试/生产环境迁移脚本
- 数据回填策略

### 7.2 测试仍远远不够

如果目标是“完全复刻”，必须补：

- API contract tests
- workflow integration tests
- database migration tests
- auth/cookie tests
- webhook retry tests
- phone callback tests
- chat tool-call tests
- customer resolver tests

没有这些测试，当前仍只能说“功能很接近”，不能说“完全复刻”。

### 7.3 需要一次真实联调回归

建议至少做一轮矩阵验证:

- 每个 TS API 路径一一对照 Python 响应
- 每个核心前端页面一一走通
- 每个关键工作流一一验证

建议按以下顺序:

1. 认证与设置
2. 会话、消息、客户、工单
3. 知识库与 AI chat
4. 自动化、标签、转派、宏
5. 渠道接入
6. Webhook、Campaign、Flow
7. 报表、导出、实时订阅

下面这份清单可以直接当成后续开发顺序。

- 完整验证 `resolve_customer` 与 TS `customer-resolver.ts` 的行为一致
- 完整闭环 `Tag / ConversationTag` 的写入、读取、导出、宏执行
- 完成 Webhook delivery 异步执行与 retry worker
- 补齐真实 Email/WhatsApp/SMS/Telegram/Phone 渠道运行链路
- 生成并验证 Alembic 初始迁移

### P1: 强烈建议完成，否则仍会有业务行为偏差

- 补上线部署手册与运行手册

## 9. 最终结论

当前 `backend` 已经不再是“原型骨架”，而是一个完成度很高的 Python 版后端迁移实现。

但如果标准是:

- “前端主要页面可跑”
  - 基本可以

- “绝大部分 API 路径存在且契约接近”
  - 基本可以

- “与原 TS 后端在功能、工作流、副作用、数据库语义、外部集成上完全等价”
  - 还没有

要真正达到“完全复刻实现原 TS 代码逻辑”，上面第 8 节的 P0 项需要全部完成，随后再完成 P1 的运行态能力补齐，最后再用一轮系统化测试和联调把结果验证出来。
